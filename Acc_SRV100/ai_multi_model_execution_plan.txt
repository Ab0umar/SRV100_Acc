# خطة تنفيذ مشروع Web App واحد فوق نظامين منفصلين

## 0. الفكرة الأساسية

المشروع عبارة عن Web App واحد للمستخدم، لكنه داخليًا مقسوم إلى نظامين منفصلين:

1. Accounting Module
   - قاعدة البيانات: SQL Server
   - الحسابات، الإيصالات، الخدمات، الإيراد، الطاقم، الليزك
   - المصدر الأساسي للحسابات يظل SQL Server

2. Medical Module
   - قاعدة البيانات: MySQL
   - الملف الطبي، الزيارات، الفحوصات، الروشتات، الملاحظات الطبية
   - المصدر الأساسي للبيانات الطبية يظل MySQL

3. نقطة الربط الأساسية
   - في SQL Server يوجد: PAT_CD
   - في MySQL يوجد: patient.id كمفتاح داخلي فقط
   - الربط الحقيقي يكون عبر حقل داخل جدول المرضى في MySQL مثل:
     - patientCode
     - أو mssqlPatientCode
   - المعادلة الصحيحة:
     SQL.PAT_CD = MySQL.patient.patientCode
   - المعادلة الممنوعة:
     SQL.PAT_CD = MySQL.patient.id

---

# 1. أهم قاعدة في المشروع

لا تخلط الحسابات مع الطبي.

يعني:

- الحسابات تقرأ وتكتب من SQL Server فقط.
- الطبي يقرأ ويكتب من MySQL فقط.
- الواجهة واحدة، لكن قواعد البيانات منفصلة.
- PAT_CD هو جسر الربط فقط.
- MySQL patient.id لا يستخدم نهائيًا للربط مع الحسابات.
- أول مرحلة في الحسابات تكون Read Only فقط.
- لا نبدأ كتابة إيصالات أو حركات مالية من الويب الجديد إلا بعد ما القراءة والربط يثبتوا تمامًا.

## المرحلة 3: تنفيذ خدمة إنشاء المريض الموحدة

المسؤول:
- Backend Agent

المخرجات المقترحة:

server/modules/patients/createPatient.service.ts
server/modules/patients/patientMapping.service.ts
server/routes/patients.routes.ts

الفكرة:

عند إنشاء مريض جديد من الويب:

1. يتم إنشاء المريض في SQL Server أولًا.
2. SQL Server يرجع PAT_CD.
3. يتم إنشاء المريض في MySQL ثانيًا.
4. يتم حفظ PAT_CD داخل MySQL في patientCode أو mssqlPatientCode.
5. لو MySQL فشل بعد نجاح SQL، لا يتم تجاهل الخطأ.
6. يتم تسجيل حالة pending sync أو retry.

Flow الصحيح:

Create Patient Request
        ↓
Validate Input
        ↓
Create Patient in SQL Server
        ↓
Get PAT_CD
        ↓
Create Patient in MySQL with patientCode = PAT_CD
        ↓
Return unified response

سيناريوهات الفشل:

1. SQL فشل:
   - لا يتم إنشاء شيء في MySQL.
   - يرجع خطأ واضح.

2. SQL نجح وMySQL فشل:
   - لا نعتبر العملية ناجحة بالكامل.
   - نسجل pending sync.
   - يظهر تنبيه أو status واضح.
   - يتم تجهيز retry لاحقًا.

3. MySQL نجح لكن response فشل:
   - لازم يكون فيه idempotency أو check يمنع التكرار.

شروط القبول:
- كل مريض جديد له PAT_CD في SQL.
- كل مريض جديد له patientCode في MySQL.
- patient.id في MySQL لا يساوي PAT_CD بالضرورة ولا يعتمد عليه.
- أي فشل يظهر ولا يتم دفنه.
